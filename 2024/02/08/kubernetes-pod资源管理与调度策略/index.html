🎯 kubernetes pod资源管理与调度策略详解

Kubernetes作为目前最流行的容器编排平台，其核心功能之一就是高效地管理和调度Pod资源。合理的资源管理和调度策略能够提高集群资源利用率，确保应用程序稳定运行。本文将深入探讨Kubernetes中Pod的资源管理与调度策略。

📊 Pod资源管理基础

资源请求与限制

Kubernetes的资源管理主要通过两个概念实现：资源请求(requests)和资源限制(limits)。

apiVersion: v1
kind: Pod
metadata:
  name: resource-demo
spec:
  containers:
  - name: app
    image: nginx
    resources:
      requests:
        memory: "64Mi"
        cpu: "250m"
      limits:
        memory: "128Mi"
        cpu: "500m"
  • requests：容器需要的最小资源量，Scheduler根据这个值来决定将Pod调度到哪个节点
  • limits：容器可以使用的最大资源量，超过这个限制可能会被终止或限流

当Pod的资源用量超过limits时：

  • 对于CPU：容器会被限流，但不会被终止
  • 对于内存：如果容器持续使用超过限制的内存，可能会被OOM Killer终止

QoS类别

基于requests和limits的设置，Kubernetes将Pod分为三个QoS(Quality of Service)类别：

  1. Guaranteed：每个容器同时设置了requests和limits，且requests等于limits
    resources:

requests:
memory: “128Mi”
cpu: “500m”
limits:
memory: “128Mi”
cpu: “500m”:hexoPostRenderEscape–>

  1. Burstable：至少一个容器设置了requests，但不满足Guaranteed的条件
    resources:

requests:
memory: “64Mi”
cpu: “250m”
limits:
memory: “128Mi”
cpu: “500m”:hexoPostRenderEscape–>

  1. BestEffort：没有任何容器设置requests或limits

QoS类别决定了Pod在资源紧张时被驱逐的优先级：BestEffort > Burstable > Guaranteed。

🧩 Pod调度策略

调度过程

Kubernetes的调度过程主要包括两个阶段：

  1. 过滤(Filtering)：筛选出满足Pod资源需求的节点
  2. 评分(Scoring)：为过滤后的节点进行打分，选择最佳节点

调度器如何工作

  1. 获取待调度的Pod
  2. 获取所有可用节点
  3. 过滤不满足Pod要求的节点
  4. 对剩余节点评分
  5. 选择得分最高的节点
  6. 将Pod绑定到选中的节点

节点选择器

通过nodeSelector可以指定Pod只能运行在特定标签的节点上：

apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  nodeSelector:
    disktype: ssd
  containers:
  - name: nginx
    image: nginx

节点亲和性

相比nodeSelector，nodeAffinity提供了更灵活的节点选择机制：

apiVersion: v1
kind: Pod
metadata:
  name: nginx-affinity
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/e2e-az-name
            operator: In
            values:
            - e2e-az1
            - e2e-az2
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 1
        preference:
          matchExpressions:
          - key: another-node-label-key
            operator: In
            values:
            - another-node-label-value
  containers:
  - name: nginx
    image: nginx

nodeAffinity有两种类型：

  • requiredDuringSchedulingIgnoredDuringExecution：硬性规则，必须满足才能调度
  • preferredDuringSchedulingIgnoredDuringExecution：软性规则，优先考虑但不强制要求

Pod亲和性与反亲和性

Pod亲和性(podAffinity)和反亲和性(podAntiAffinity)允许根据已经在节点上运行的Pod标签来约束Pod的调度：

apiVersion: v1
kind: Pod
metadata:
  name: redis-pod
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - web
        topologyKey: "kubernetes.io/hostname"
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchExpressions:
            - key: app
              operator: In
              values:
              - redis
          topologyKey: kubernetes.io/hostname
  containers:
  - name: redis
    image: redis
  • podAffinity：与特定标签的Pod调度在同一位置
  • podAntiAffinity：避免与特定标签的Pod调度在同一位置

污点(Taints)和容忍(Tolerations)

污点允许节点排斥一组Pod，而容忍允许Pod调度到带有匹配污点的节点上。

为节点添加污点：

kubectl taint nodes node1 key=value:NoSchedule

在Pod定义中添加容忍：

apiVersion: v1
kind: Pod
metadata:
  name: nginx-toleration
spec:
  tolerations:
  - key: "key"
    operator: "Equal"
    value: "value"
    effect: "NoSchedule"
  containers:
  - name: nginx
    image: nginx

污点效果包括：

  • NoSchedule：不允许调度新Pod（除非有容忍）
  • PreferNoSchedule：尽量避免调度新Pod
  • NoExecute：驱逐没有匹配容忍的现有Pod

🔍 资源分配高级特性

Resource Quota

ResourceQuota用于限制命名空间内的资源总量：

apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-resources
  namespace: dev
spec:
  hard:
    pods: "10"
    requests.cpu: "4"
    requests.memory: 4Gi
    limits.cpu: "8"
    limits.memory: 8Gi

Limit Range

LimitRange为命名空间中的Pod和容器设置默认资源限制：

apiVersion: v1
kind: LimitRange
metadata:
  name: limit-range
  namespace: dev
spec:
  limits:
  - default:
      cpu: 500m
      memory: 512Mi
    defaultRequest:
      cpu: 200m
      memory: 256Mi
    type: Container

垂直Pod自动伸缩(VPA)

VPA可以自动调整Pod的CPU和内存请求：

apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
spec:
  targetRef:
    apiVersion: "apps/v1"
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: '*'
      minAllowed:
        cpu: 100m
        memory: 50Mi
      maxAllowed:
        cpu: 1
        memory: 500Mi

🚀 实际应用中的最佳实践

1. 合理设置资源请求和限制

  • 基于应用实际需求设置requests和limits
  • 监控应用实际资源使用情况，逐步调整
  • 对关键应用设置为Guaranteed QoS类别
resources:
  requests:
    memory: "1Gi"
    cpu: "500m"
  limits:
    memory: "1Gi"
    cpu: "500m"

2. 使用节点亲和性实现高可用

确保Pod分布在不同可用区，提高系统可用性：

affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - key: app
          operator: In
          values:
          - web
      topologyKey: "topology.kubernetes.io/zone"

3. 利用污点和容忍分离工作负载

使用污点将特定工作负载(如日志收集、监控)限制在专用节点：

# 为特定节点添加污点
kubectl taint nodes monitoring-node dedicated=monitoring:NoSchedule

# 在监控Pod中添加匹配的容忍
tolerations:
- key: "dedicated"
  operator: "Equal"
  value: "monitoring"
  effect: "NoSchedule"

4. 设置Pod中断预算(PDB)

确保在节点维护时应用保持最小可用实例：

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: web

5. 适当使用优先级和抢占

为关键应用设置更高优先级：

apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000
globalDefault: false
description: "高优先级Pod"
---
apiVersion: v1
kind: Pod
metadata:
  name: critical-app
spec:
  priorityClassName: high-priority
  containers:
  - name: app
    image: critical-app

📝 总结

Kubernetes的资源管理与调度策略是一个复杂而强大的系统，通过合理配置：

  1. 资源请求和限制可以确保Pod获得所需资源
  2. QoS类别提供了资源压力下的优先级保障
  3. 亲和性规则能够优化Pod的布局和高可用
  4. 污点和容忍可以实现专用节点和混合工作负载
  5. ResourceQuota和LimitRange帮助管理多租户环境

掌握这些策略，可以构建更加稳定、高效的Kubernetes集群，为应用提供更好的运行环境。随着Kubernetes的发展，调度器也在不断优化，提供更多灵活的调度策略，满足更复杂的业务需求。
